📋 What's happening

Apple removes the Liquid Glass opt-out flag in iOS 27 (mid-September 2026), making the new material mandatory. Instacart's iOS app was critically broken on initial testing. The working team is iOS-centric — compatibility burndown in Q3, nav adoption in Q4. LG effectively ends the era of OS-neutral design systems, requiring teams to separate native OS interfaces from the underlying brand and product layer in a way that wasn't necessary before.

The internal doc is a solid iOS playbook. It has no answer for web, Android, or the shared design system library.

🍎 Apple Liquid Glass — Key Points

What it is

  • Real-time light bending (lensing) — refracts content behind UI chrome
  • Specular highlights — respond to device motion via IMU
  • Fluid morphing — components transform between states like liquid
  • Hardware-dependent — requires Apple Silicon + Metal GPU pipeline
  • System-wide across iOS, iPadOS, macOS, watchOS, tvOS

Critical HIG rules

  • Navigation layer only — never apply to content (lists, tables, media)
  • iOS 27 reduced default transparency after beta pushback
  • iOS 27 added a user-facing glass vs. tint slider in Settings
  • Reduced transparency + high contrast must degrade to solid surfaces
  • Apple-platform only — no official web or Android equivalent

A11y risk — known issue, baked into the material

  • iOS 26 beta had low-contrast text over heavy blurs — Apple received significant feedback and dialed back
  • WCAG AA over frosted backgrounds requires tight contrast management + solid fallbacks
  • Reduced Transparency / Increase Contrast accessibility settings must trigger opaque surface fallback
  • This is a design system responsibility — it has to be enforced at the component level, not left to product teams

⚠️ Gaps from my POV

The plan as written is an iOS execution plan. These are the gaps the team needs to address — or explicitly acknowledge as deferred with a revisit date.
Gap Why it matters Status in doc
Web platform strategy Instacart.com users won't see LG. Do we approximate, intentionally diverge, or say nothing? Missing entirely
Android strategy Material You is a different visual language. iOS gets LG nav — Android users see something completely different. Parity question is open. Missing entirely
Android navigation patterns Beyond styling, iOS and Android have structurally different nav paradigms. LG directly targets nav components (tab bars, nav bars) — but Android uses system-level back gestures, no persistent tab bar standard, and Material You nav conventions that are behaviorally incompatible with iOS patterns. This is a UX pattern gap — behavioral and structural, deeper than styling. Shipping "Android parity" of LG nav components may not be meaningful or correct. Not addressed
Multi-style library governance One Pantry with LG + web + Android variants? Or fork? Who owns the token layer? Not addressed
Component drift risk iOS LG components evolve quarterly. Web/Android lag. Without a strategy, divergence compounds silently. Not addressed
A11y compliance LG had known contrast issues in early betas, but iOS provides system-level settings (Reduce Transparency, Increase Contrast) that render surfaces fully opaque — which may largely address this at the OS level. Worth validating how much app-level work actually remains before treating this as a blocker. Note: that OS-level safety net doesn't exist on web or Android, so any glassmorphism approximation on those platforms places the a11y burden on the component itself. Mentioned, no owner
Token architecture LG needs semantic tokens for depth, translucency, blur. Do web/Android inherit them? Who designs this? Implied, not designed
Brand expression collision H2 Brand updates run concurrently. If both brand and LG touch tokens simultaneously, web and Android absorb a double disruption. Risk flagged, not resolved

💡 My recommendations

01
Align on design principles across platforms
Extract the design intent behind LG — depth through translucency, hierarchy via material, fluid interaction — and codify it as platform-agnostic principles. iOS expresses this through LG. Web through backdrop-filter and layering. Android through Material You elevation and tonal surfaces. This extends to navigation patterns too: Android uses system back gestures, drawer navigation, and its own bottom nav conventions. Android nav components should follow Android standards and express the same underlying design intent through those native patterns.
Web Android Long-term
02
Token-first: build the LG semantic layer now
Before iOS components ship, define semantic tokens for depth, translucency, blur strength, surface opacity. Prevents hardcoded values in iOS components. Web and Android consume the same token intent — each platform resolves it natively. This is the systemic lever that stops drift from accumulating silently.
IDS Web Q3 priority
03
Component parity matrix — make drift visible
A living matrix tracking each target component's state across iOS, web, and Android: native LG / approximated / intentionally diverged / not started. Source of truth for quarterly alignment. Explicit divergence decisions are fine — silent drift is the enemy.
Governance Low cost, high signal
04
Own a11y as a system-level constraint
LG has a proven contrast problem. IDS should be the guardrail: every LG component ships with a tested reduced-transparency solid fallback, documented contrast ratios, and reduced-motion behavior. This belongs at the design system level, enforced through components.
A11y WCAG AA All platforms
05
Position web as the stable brand canvas
iOS and Android are both mediated by the OS — the OS applies materials, nav conventions, and interaction patterns on top of your product. Web has none of that. There's no system material layer, no enforced nav pattern. What you ship is exactly what the user sees. That makes web the one platform where brand expression is fully under our control. Rather than framing web as the platform playing catch-up to LG, it can be the reference point — the place where Instacart's design language lives in its purest form, without OS mediation. iOS and Android then adapt that intent to their native constraints. This reframes the web question from "how do we approximate LG" to "what does our design language look like when it's fully ours."
Web Brand Strategic framing

On the library question: one library, multiple modes

Forking Pantry into an "iOS LG library" and a "Web library" creates two maintenance burdens and will diverge fast. The better model: one Pantry library with explicit platform modes — like dark/light mode today. Each component has a "LG mode" (iOS), "glass approximation" (web), and "material" (Android) variant, all sourced from the same token base. The token layer is the contract. Platform implementations are the interpreters.

The key question to land in this meeting: Does the team agree that design system owns the cross-platform principle layer — or is web/Android each team's own problem? That decision determines everything downstream.

❓ Questions I'm bringing to the room

  • Web components: When iOS nav adopts LG in Q4, what is the web nav doing? Intentional divergence or planned follow-up? Who decides and when?
  • Android: Are we explicitly accepting that Android will look significantly different from iOS post-LG? If yes, that's a product decision PMs need to own — not just design systems.
  • Android nav patterns: LG targets nav components specifically, but Android navigation is structurally different — system back gesture, no universal tab bar standard, drawer patterns. Do we even want Android nav to mirror iOS? Or should Android follow Material You nav conventions independently, and we align on intent rather than structure?
  • Library strategy: Is IDS expected to maintain LG + non-LG component variants going forward? What's the deprecation path when LG becomes the iOS default?
  • Token ownership: Who is designing the semantic tokens that underpin LG material properties? Needs to be scoped before components are built.
  • Brand expression collision: H2 brand updates run concurrently. How do we sequence so LG and brand changes don't create conflicting token updates?
  • Adoption definition: "2027 soft requirement" — does that mean iOS only, or does web/Android also need to express the new design language by 2027?
  • A11y ownership: Who validates WCAG AA compliance for each LG component before it ships? Is there a shared QA protocol across iOS, web, and Android?
